AI options for Tincture of Mercy

The best direction is not “LLMs drive the game.” It is: deterministic game AI owns action, local models add memory, language, texture, accessibility, and tooling where they can be bounded, cached, and verified.

How AI should work in our game

Game AI is usually a stack: perception reads world state, decision logic chooses an action, movement/timers execute it, and the simulation validates consequences. For Tincture, that maps cleanly to the substrate already being built: an AI layer should read projections and event history, then submit a normal verb request. It should not mutate resources, death, loot, notebook, or progression directly.

1. PerceiveActor state, spatial context, boy position, wounds, noise, prior events.
2. Score/planUtility, behavior tree, or GOAP selects a candidate action.
3. ProposeAI emits a typed intent: verb, target, urgency, reason key.
4. ValidateVerbInvocation checks costs, cooldowns, timers, item requirements.
5. ResolveModifierAssembler + OutcomeResolver append authoritative events.
6. ProjectPresenters/notebook/debug views interpret the same event truth.
Design guardrail: local models can author or rank suggestions, but the C# substrate decides what happened. This preserves replay, testing, and the shared care/combat substrate.

Industry-standard AI

Finite state machines / HFSM: simple, reliable phase control for animation, attack windups, recovery, fleeing, and patient bedside states.

Behavior trees: readable authored behaviors for wolves, companions, patients, and scripted teaching encounters.

Utility AI: scores actions by hunger, fear, proximity, wound severity, boy danger, pressure, and opportunity. This is the best fit for emergent care/combat tension.

Planning AI

GOAP / HTN: good for longer action chains: villager routines, pack coordination, pilgrimage logistics, patrols, retreat, call-for-help, or “find safer route then act.”

Use later, once B4 has spatial/route primitives. It is powerful but more expensive to author and debug than utility + behavior trees.

Model-assisted AI

Embeddings: cheap local semantic matching for memory, receptivity, hints, barks, and notebook retrieval.

Small LLMs: constrained prose, recap, emotional barks, optional dialogue, and dev tools. Keep outputs schema-limited and non-authoritative.

Innovative low-cost ideas

  • Witness memory without hallucination: embed important events and retrieve them later for notebook/progression copy; no model decides truth.
  • Receptivity shadows: use semantic similarity to choose which path/register framing an NPC is most ready to hear.
  • Pressure director: utility system modulates danger, rest windows, and call/flee behavior from event history.
  • Boy as attention beacon: simple utility signals make the child tactically meaningful without needing a complex companion brain.
  • Constrained local barks: tiny model drafts 1-line reactions, then style/allowlist/filter/cache before display.
  • Playtest confessor: dev-only model summarizes replay failures in plain language after test runs.

Recommended adoption path

  1. B4 now: build spatial context + threat utility + leash/respawn. No model required.
  2. B5d/B6: add debug presenters so AI decisions are explainable from event truth.
  3. Prototype A: local embeddings for memory/receptivity/hint retrieval.
  4. Prototype B: tiny local LLM for optional cached barks and notebook recap drafts.
  5. Tools lane: cloud or local model reads replay fixtures for design QA.
Decision: start with deterministic AI because it makes combat, care, and replay stronger. Add tiny models at the edges where their uncertainty is useful rather than dangerous.

Local runtime candidates

llama.cpp / GGUF: broad hardware support, quantization, local server mode, and C# bindings through LLamaSharp. Best native route for Godot desktop experiments.

ONNX Runtime: good for embeddings/classifiers in C# when the model is small and fixed.

WebLLM / Transformers.js: useful for browser companion tools or web demos; less direct for Godot runtime.

Ollama / LM Studio: fastest development loop, but not ideal as a shipped dependency.

Model fit quick take

Embeddings: all-MiniLM-L6-v2 or similar small embedding model; good for semantic retrieval and clustering.

Text tiny: SmolLM2 360M for very small experiments; Qwen3-0.6B for stronger small dialogue/tool behavior; Gemma 3 270M/1B for Google open-weight line.

Text better: 3B–4B class models can improve quality but raise memory, latency, and packaging burden.
